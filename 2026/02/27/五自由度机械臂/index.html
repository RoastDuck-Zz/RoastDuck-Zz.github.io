一句话总结： 用不到一千块的硬件成本，从零造了一台会自己「看」焊缝、自动追踪焊接的五自由度机械臂。

# 为什么要做这个？

大四毕业设计，我选的题目是「五自由度关节型焊接机器人结构设计与控制系统设计」。听起来很教科书，但背后的动机很实在：

市面上稍微像样的工业焊接机器人动辄十几万，对中小企业或者个人工坊来说门槛太高。我就想试试——能不能用消费级的硬件，搭出一台具备基本视觉引导能力的焊接机械臂？

带着这个问题，我花了几个月时间，从 SolidWorks 画图开始，一步步走到了 YOLOv8 视觉闭环控制。这篇文章把整个过程记录下来，重点讲我踩过的坑、用到的技术方案，以及一些收获。

# 整体思路：五轴怎么分配？

工业上常见的焊接机器人是六轴的。六轴的好处是末端姿态有更大的冗余空间，但代价是——多一个关节意味着多一套舵机和传动结构，控制算法的复杂度也是指数级上升。

仔细分析了焊接这个场景后，我发现：焊枪的姿态其实不需要六轴那么大的冗余。只要能保证焊枪垂直于焊缝表面（调整推拉角），末端有效覆盖工作空间就够了。于是我把方案精简成了五轴：

关节 作用 类比
J1 基座水平回转 腰
J2 大臂俯仰 肩
J3 小臂俯仰 肘
J4 腕部俯仰 腕
J5 末端自旋 手

五个自由度在 200mm 工作半径内能覆盖所有必要姿态，又比六轴少了一套驱动，省钱也省心。

# 机械结构：SolidWorks + 3D 打印的反复拉锯

机械臂的骨架是整个系统的物理基础。我用 SolidWorks 从零画了全套三维模型——基座、第一连杆、主承力连杆（大臂+小臂）、末端快换接口。

# 材料怎么选？

3D 打印的常用材料就那几种。我专门做了个对比：

材料 弯曲强度 热变形温度 结论
普通 PLA 较高 极低 靠近焊接热源会软化，一票否决
ABS 中等 较高 收缩率大，打出来电机安装孔都对不准
9600 高韧性树脂 极高 59°C 光固化精度极高，耐热阈值够用，选了它

最终用了 SLA 光固化 9600 树脂，通过嘉立创的 3D 打印服务下单。树脂材料打出来的件表面特别细腻，装配公差控制在了一个非常舒服的范围。

# 设计上的一些关键考量

基座： 直径做得比较大，阶梯圆盘式。因为整条手臂伸展出去时基座承受的力矩最大，底盘不稳后面都是白搭。

第一连杆高度 39mm： 这个数字不是拍脑袋定的。太矮了——大臂低头时会撞到基座上的电子舱；太高了——重心上移，基座关节要多克服额外的重力矩。39mm 是我在 SolidWorks 里反复做干涉检查后确定的平衡值。

埋线设计： 每个连杆内部都预留了走线孔和理线槽。这条经验来自之前做其他项目的教训——多轴运动时电线缠绕是很头疼的问题。把线藏在骨架里面，外观干净，物理上也杜绝了快速转动时扯断信号线的风险。转角处全部做了圆角处理，减少 3D 打印应力集中的问题。

末端执行器： 做成了模块化的快拆接口。现在装的是模拟焊枪，以后换爪头、吸盘都很方便，不用重新设计整条手臂。

# 硬件方案：ESP32-S3 凭什么当主控？

单片机选型这件事，我在 Arduino UNO、STM32、ESP32-S3 三者之间纠结了很久。

# 为什么不用 Arduino UNO？

ATmega328P，8 位架构，主频才 16MHz，SRAM 只有 2KB。驱动几个舵机还行，但想让它同时处理 OV3660 摄像头的图像数据？门都没有。连 DVP 接口都没有。

# STM32 呢？

STM32F4/H7 系列是工业级的标杆。定时器精度高，PWM 输出稳，驱动舵机是一把好手。但问题有两个：一是自带 Wi-Fi 的型号少且贵，想无线传图还得外挂模块，硬件复杂度直接翻倍；二是开发环境（HAL 库、时钟树配置）学习曲线陡峭，对于毕设周期来说时间成本太高。

# ESP32-S3 的取舍

最后选了 ESP32-S3-DevKitC-1。Xtensa 32 位 LX7 双核，主频 240MHz，自带 Wi-Fi/BLE 和 DVP 摄像头接口，关键是一片才几十块。它的向量指令集对神经网络推理有加速作用——虽然主要 AI 推理放上位机，但端侧做一些轻量预处理时很有用。

说实话，ESP32-S3 在极端环境下的硬件定时器精度不如 STM32，但对于桌面级的焊接场景完全够用。这个价格有这些功能，性价比真的离谱。

# 舵机配置：非对称驱动的逻辑

五轴都用一个型号的舵机当然方便，但力学上不合理。

基座、大臂、小臂（J1-J3）：MG996R 金属齿轮舵机，13kg·cm 扭矩。这几个关节需要承载整机重力矩和动态冲击，扭矩不够会直接抖。

腕部自旋（J5）：SG90 微型舵机，仅 9g 重。末端轻量化有什么好处？转动惯量小。J2、J3 不用额外对抗末端的惯性载荷，整条手臂的动态响应速度明显提升。

中间的 J4（腕部俯仰）用了另一颗 MG996R——这个关节需要承受焊枪和摄像头的重量，SG90 扛不住。

# PWM 驱动：为什么要加 PCA9685？

ESP32-S3 本身能输出 PWM，但引脚数量有限。更关键的是——主控要同时处理 Wi-Fi 图像透传、运动学逆解、PID 计算，根本没精力一直维持 PWM 波形。

PCA9685 解决了这个问题。它通过 I²C 总线（只要两根线 SDA/SCL）就能输出 16 路 PWM，内置独立时钟。主控下完角度指令后就能去做别的事，PCA9685 自己维持波形。这是一种硬件级任务卸载。

驱动频率统一设定为 50Hz，0°-180° 映射到 150-600 的脉宽计数值。12 位分辨率（4096 级）对于舵机的微调完全够用。

# 供电隔离：差点重启到崩溃

多颗 MG996R 同时在重载下换向时，瞬间电流非常大。如果把舵机电源和主控逻辑电源共用一路，电压波动会直接导致 ESP32-S3 低压复位——Wi-Fi 断开、程序重跑。

解决方案是 XL4015 双通道降压模块，物理隔离两路供电：

  • 动力链路：给舵机群供电，5A 额定，扛得住多轴齐动时的瞬态功率
  • 逻辑链路：给 ESP32-S3、PCA9685 逻辑部分、OV3660 传感器供电，稳压滤波

这个方案也是踩坑踩出来的——之前调试时经常莫名其妙重启，排查了半天才发现是电源纹波导致的。

# 运动学：手算逆解的快乐

有了机械结构，下一步就是用数学描述它的运动。

# D-H 建模

我采用改进的 D-H 参数法，在相邻连杆间建立 4×4 齐次变换矩阵。每个关节固连一个空间直角坐标系，然后把物理尺寸代入：

  • l₀ = 28mm：基座中心到肩关节轴线的水平偏移
  • d₁ = 85mm：基座旋转轴垂直高度
  • l₂ = 105mm：大臂有效杆长
  • d₄ = 100mm（原设121mm，建模优化后调整）：小臂有效杆长
  • d₆ = 50mm：末端法兰到腕关节偏移

正运动学就是给定五个关节角度，算出末端位姿。这一步相对直接。

# 逆运动学：为什么要手推解析解而不是数值迭代？

逆运动学是从目标位姿反推关节角度。有两种主流方法：

  1. 数值迭代法：类似牛顿法逐步逼近，通用但不保证收敛，而且每次迭代都要算矩阵乘法
  2. 解析法：利用几何关系直接推出封闭公式，一次算出结果

我选了投影几何解析法。原因是 ESP32-S3 要同时处理 Wi-Fi 透传和中断响应，算力不能全砸在运动学上。数值迭代在单片机上的执行时间不可预测——有时候收敛快，有时候要迭代十几轮，这会影响闭环控制的实时性。

解析法的推导过程：

第一步——求基座角 q₁： 把末端目标点投影到水平面，用 atan2(y, x) 直接算，一个三角函数搞定。

第二步——求大臂和小臂（q₂, q₃）： 这是最有意思的部分。把三维问题降维到二维——投影到由连杆 l₂ 和 d₄ 构成的垂直平面。然后用余弦定理：已知肩到腕的直线距离 s，以及两段连杆长度 l₂ 和 d₄，直接算出肘关节的内角。再加上投影的仰角分量，q₂ 和 q₃ 就出来了。

第三步——求末端姿态（q₄, q₅）： q₄ 根据目标俯仰角和已算出的 q₂+q₃ 反推。q₅ 单独由焊缝切向方向算偏转角。

整个逆解过程在 ESP32-S3 上做到了微秒级输出，代价就是推导时在草稿纸上画了无数个三角形。

学习心得： 数学课上学余弦定理时觉得"这玩意儿能有什么用"，真到做项目才发现，把三维空间投影到二维平面然后用余弦定理，是最优雅的解法。工程实践是理论最好的老师。

# 视觉系统：YOLOv8 为什么比传统方法强？

焊接环境对视觉算法极其不友好——电弧强光闪烁、飞溅的火花、弥漫的烟尘。如果用传统的 OpenCV 方案（灰度化→高斯滤波→Canny 边缘检测→霍夫变换），在这种环境里基本就是白给：强光会让二值化阈值瞬间失效，烟尘会制造大量假边缘。

# 引入 YOLOv8

YOLOv8 是单阶段的端到端目标检测模型，不需要人工设计特征。模型自己从数据中学习焊缝的深层语义模式——即使部分被烟尘遮挡，也能靠上下文特征推断出焊缝位置。

选择 YOLOv8 而不是其他模型的原因很简单：

  • 轻量化版本在保证精度的同时推理速度够快
  • Ultralytics 库生态好，训练部署一条龙
  • 社区活跃，踩坑了有人可问

# 数据集：训练不能没有好数据

工业焊缝的开源数据集非常少。最后我在 Roboflow 平台上找到了一套高质量的焊缝检测数据——包含 V 型坡口、搭接、T 型接头等多种接头形式，覆盖了不同光照和烟尘条件下的场景。一共 838 张标注好的图片。

但 838 张对于训练深度神经网络远远不够，容易过拟合。我写了个 Python 脚本做离线数据增强：

  • 几何变换：随机水平/垂直翻转、限定角度旋转——模拟机械臂从不同位姿观察焊缝的视角变化
  • 像素变换：高斯噪声——模仿焊接现场的金属粉尘和传感器噪声；亮度/对比度随机调整——模拟电弧起灭产生的极端光照变化

增强后数据集扩展到 3352 张，按 8:1:1 随机拆分成训练集、验证集、测试集。训练时还用了 YOLO 自带的 Mosaic 增强——把四张不同场景的图随机拼接在一起训练，强制模型学习更多样化的背景特征。

# 训练过程

在 PyTorch 上跑，GPU 加速。关键参数：

参数 值 说明
输入尺寸 640×640 平衡细节与速度
Batch size 16 GPU 内存允许的范围
初始学习率 0.01 SGD 优化器
迭代轮次 100 约 80 轮后稳定

从 Loss 曲线来看，定位损失和分类损失在前 80 轮平稳下降后就趋于稳定，没有出现过拟合也没有梯度爆炸——数据增强起了很大作用。

# 检测效果

最终模型在测试集上的成绩：

  • 精确率 94.2%：预测为焊缝的框里，有 94.2% 确实是对的
  • 召回率 91.5%：真实焊缝中，有 91.5% 被模型找到了
  • mAP@0.5 达 95.8%：综合指标，说明模型在各种背景下都能稳定检出焊缝

从混淆矩阵看，模型把背景误判为焊缝的概率极低——这对于后续 PID 控制非常关键，因为如果出现假阳性，机械臂会追着一个不存在的东西跑。

学习心得： 做深度学习项目，80% 的时间花在数据上。数据集的多样性比模型参数的微调重要得多。之前调了半天模型 mAP 就是上不去，后来发现是训练集里缺少烟尘场景——加上增强策略后效果立竿见影。

# 视觉伺服：从「看到」到「走到」

检测到焊缝只是第一步，接下来要让机械臂追着焊缝走。

# 为什么选 IBVS 而不是 PBVS？

传统做法是基于位置的视觉伺服（PBVS）——先做手眼标定算出相机和机械臂基座的变换矩阵，再把图像坐标映射到三维空间。但问题是：标定精度高度依赖硬件安装精度，微型摄像头稍微有点位移或者 3D 打印件有微小形变，累积误差就会大到影响焊缝跟踪。

我选了基于图像的视觉伺服（IBVS）——直接在二维图像平面里闭环，完全跳过手眼标定。逻辑很简单：维持焊缝中心在图像视野正中央。如果焊缝偏左了，就控制机械臂往左移，直到偏差归零。

# 像素级特征提取

YOLOv8 输出焊缝的边界框坐标。我取边界框的几何中心作为引导点：

cx = (x1 + x2) / 2
cy = (y1 + y2) / 2

然后用低通滤波消除帧间噪声——避免因为某一帧的抖动让机械臂做出过度反应。

# 增量式 PID 控制器

控制目标是让焊缝中心锁定在图像中心点 (320, 240)。计算当前焊缝中心与靶心的像素偏差 e_x, e_y，输入双轴 PID 控制器，输出机械臂在 X/Y 方向的期望补偿速度。

核心代码逻辑：

error = target - current
integral += error * dt
integral = clamp(integral, -1000, 1000)  // 抗积分饱和
derivative = (error - prev_error) / dt
output = Kp*error + Ki*integral + Kd*derivative

经过反复调参，最终锁定 Kp=0.04, Ki=0.001, Kd=0.015。当像素偏差落入预设死区（约 ±5 像素），期望速度归零，机械臂进入锁定状态。

积分限幅（Anti-Windup） 很重要——焊接过程中机身震动会产生微小超调，如果积分项不断累积，会导致系统越来越不稳定。加了限幅后，即使偶尔有较大偏差，系统也能快速恢复。

# 端云协同

这是整个系统架构的精髓：

  1. ESP32-S3 端：负责图像采集（OV3660→DVP→Wi-Fi 透传）、接收上位机指令、通过 I²C 驱动 PCA9685 输出 PWM、执行舵机运动
  2. 上位机（PC）端：运行 YOLOv8 推理、计算像素偏差、跑 PID 控制器、解算速度级逆运动学、通过 UDP 下发关节速度指令

为什么要这样分？因为 AI 推理需要算力——ESP32-S3 虽然有一定的神经网络加速能力，但跑 YOLOv8 的完整推理还是吃力。把推理放到上位机，ESP32-S3 专职做实时控制和通信，各司其职。

通信延迟实测在 20ms 以内，闭环频率达到 15fps 以上，对桌面级焊接来说完全够用。

学习心得： 做嵌入式系统的边界很重要——知道什么该放在端侧、什么该放在云端。不是所有事情都要在单片机上做，把重型计算卸载到上位机，单片机只做它擅长的事（实时控制），这个架构思路来自实习时做 ESP32-S3 项目的经验。

# 轨迹规划：LSPB 解决启停抖动

舵机在启停瞬间很容易抖——特别是 MG996R 在承受负载时，PWM 信号突变会引发机械冲击。

我引入了 LSPB 算法（Linear Segments with Parabolic Blends）。简单说：把每次运动切成三段——抛物线加速、匀速、抛物线减速。在启停阶段用抛物线平滑过渡，中间段保持匀速。

实现时设定了 200ms 的加减速时间窗口。效果很明显——之前不加 LSPB 时机械臂停止会有一个"弹回来"的微震动，加上之后停止动作非常丝滑。这对于焊接场景尤其重要，因为末端抖动会直接影响焊缝成型质量。

# 实验结果

样机组装完成后做了两轮测试：

结构实验：实测有效工作半径达到了预期的 200mm。9600 树脂骨架在满载悬臂伸展时重心保持稳定，没有出现明显的结构抖动或丢步。

视觉伺服闭环实验：在人为移动焊缝工件位置后，系统在大约 2-3 秒内完成了重新锁定。YOLOv8 在复杂背景下（强光、烟尘模拟）依然稳定检出焊缝，mAP 保持在 95% 以上。低通滤波 + PID 的组合有效抑制了帧间抖动，末端轨迹平滑无跳变。

# 不足与改进方向

写完论文回头看，这套系统还有不少可以提升的地方：

  1. 单目 → 双目/结构光：OV3660 是单目摄像头，没有深度信息。遇到高低起伏的工件时，只能依赖二维平面的像素偏差，没法精确获取焊缝的三维坐标。升级双目视觉或微型结构光是一个明确的方向。

  2. 材料耐热：9600 树脂虽然比 PLA 强，但热变形温度毕竟只有 59°C，离真正工业级应用还差得远。后续如果要把这套方案产品化，前端连杆必须换成铝合金 CNC 或者碳纤维复合材料。

  3. 舵机 → 闭环伺服电机：普通 PWM 舵机在长时间重载下会产生热漂移，而且没有位置反馈。换成带高分辨率编码器的闭环伺服电机，配合动力学重力补偿，重复定位精度能上一个量级。

  4. 模型的泛化能力：目前 YOLOv8 模型在训练过的焊缝类型上表现很好，但遇到完全没见过的接头形式会下降。需要持续扩展数据集。

# 写在最后

这几个月从零搭一台机器人，最深的感受是——做硬件不像写代码，有 Bug 能随时改。一个 3D 打印件尺寸不对，就要重打再等几天；一个供电设计不合理，排查问题排查到怀疑人生。

但也是这个过程让我真正理解了什么是"工程化思维"——选型时的取舍、数学推导的应用、软硬件协同的边界划分。这些不是在课堂做题能学到的。

如果你也对低成本机器臂或者嵌入式视觉感兴趣，欢迎交流。所有的 SolidWorks 模型、代码和训练数据我后续会整理放出来。

毕设答辩结束，探索才刚刚开始。
